# Story 1.4: Pro Logout & Session Security — Design Spec

## Context

Epic 1 (Registration & Multi-Tenant) final story. Stories 1.1-1.3 delivered registration, OAuth2 login, and password reset. This story completes the auth lifecycle with logout, brute force protection, and route guarding.

## Decisions

- **JWT revocation:** None. Short-lived tokens + client-side cleanup. No server-side blacklist.
- **Brute force:** Database-backed (columns on `User` entity). Persists across restarts.
- **Route guard:** Created and ready, applied to `/pro` placeholder route. Real usage starts Epic 2.

## 1. Logout (Client-Side)

No backend endpoint. `AuthService.logout()` already handles:
- Clear JWT from `localStorage`
- Reset `currentUser` and `token` signals to null
- Navigate to `/`

**Header changes:**
- When authenticated: show user menu button (user icon) that opens a dropdown with user name/email and a "Logout" button
- When not authenticated: show login button (existing behavior)
- Use Angular Material `mat-menu` for the dropdown

## 2. Brute Force Protection (Backend)

**User entity — 2 new columns:**

| Column | Type | Default | Description |
|--------|------|---------|-------------|
| `failed_login_attempts` | `Integer` | `0` | Counter of consecutive failed logins |
| `account_locked_until` | `Instant` | `null` | Timestamp when lockout expires |

**Login flow changes in `AuthController.login()`:**

```
1. Find user by email (existing)
2. IF account_locked_until > now:
     → 423 LOCKED response with remaining lockout duration
3. IF password incorrect:
     → Increment failed_login_attempts
     → IF failed_login_attempts >= 5: set account_locked_until = now + 15 min
     → 401 UNAUTHORIZED response
4. IF password correct:
     → Reset failed_login_attempts = 0, account_locked_until = null
     → Return JWT (existing)
```

**Response for locked account (423):**
```json
{
  "error": "ACCOUNT_LOCKED",
  "message": "Account temporarily locked. Try again in X minutes.",
  "retryAfterSeconds": <remaining_seconds>
}
```

**Frontend handling:**
- Login component checks for 423 status
- Displays translated lockout message with remaining time

## 3. Auth Guard (Frontend)

**Functional guard (`authGuard`):**
```typescript
export const authGuard: CanActivateFn = () => {
  const authService = inject(AuthService);
  const router = inject(Router);
  return authService.checkAuthentication().pipe(
    map(authenticated => authenticated || router.createUrlTree(['/login']))
  );
};
```

**Route setup:**
- `/pro` route (lazy-loaded placeholder component) with `canActivate: [authGuard]`
- Guard reusable for all future pro/authenticated routes

## 4. Translations

**New keys added to both `fr.json` and `en.json`:**

```
auth.logout                    → "Déconnexion" / "Sign out"
auth.userMenu.greeting         → "Bonjour, {{name}}" / "Hello, {{name}}"
auth.errors.accountLocked      → "Compte temporairement bloqué..." / "Account temporarily locked..."
auth.errors.tooManyAttempts    → "Trop de tentatives..." / "Too many attempts..."
```

## 5. Tests

**Backend (JUnit 5):**
- Login with locked account → 423 response
- 5 failed attempts → account becomes locked
- Login after lockout expires → succeeds, counters reset
- Successful login resets failed attempts counter

**Frontend (Jasmine):**
- `authGuard`: redirects to `/login` when unauthenticated, allows when authenticated
- Header: logout button visible when authenticated, hidden when not
- Login component: displays lockout message on 423 response
